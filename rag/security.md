# Security Configuration (RAG-Friendly)

## Quick Answer

**How do I configure HTTP security across my microservices?**
Configure it per service using standard Spring properties targeting the `middleware.security.*` prefix. Choose an authentication type (`NONE`, `BASIC_AUTH`, `JWT`, `OIDC`, `API_KEY`) and map your `public-paths` and `protected-paths` explicitly.

**YAML Configuration (API_KEY Example):**
```yaml
middleware:
  security:
    type: API_KEY
    public-paths:
      - /actuator/health
      - /api-docs/**
    protected-paths:
      - type: ROLES
        path: /api/*/catalogs/*/products
        methods: [ GET ]
        allowed-roles: [ LIST_PRODUCTS, ADMIN ]
    api-key:
      header-name: X-API-KEY
      credentials:
        - key: ${API_KEY_VALUE:default-secret}
          enabled: true
          roles: [ LIST_PRODUCTS, ADMIN ]
```

**Constraints:**
- Setting `type: NONE` disables security completely (permit all), turning off CSRF.
- The `ROLES` defined in configuration are translated automatically into Spring Security `ROLE_` authorities (e.g. `ROLE_ADMIN`).
- An API Key rejection (e.g., incorrect or missing) leverages the same unified `ErrorMessage` JSON response format (defined in `docs/errors.md`).

## Authorization Types

### What mapping rules decide if an endpoint is public or protected?
The `ProtectedPathRule` processor iterates your HTTP request sequentially.

**Logic Flow:**
1. It checks against `public-paths` (Ant-style matching). If matched, it permits heavily.
2. It evaluates `protected-paths` checking the `type` flag:
   - `type: NONE`: Treats the matched pattern as public (`permitAll()`).
   - `type: AUTHENTICATED`: Requires valid authentication but doesn't check any roles.
   - `type: ROLES`: Must be authenticated AND have at least one defined `allowed-roles`.

**Constraints:**
- If the endpoint matches NOTHING, it defaults to standard `authenticated()` mode internally.

## Alternative Authentication Methods

### How do I configure JWT parsing and roles claim extraction?
Use `type: JWT` and specify the HMAC secret and the token JSON authority claim target.

**YAML Configuration:**
```yaml
middleware:
  security:
    type: JWT
    jwt:
      secret: ${JWT_SECRET:change-me}
      algorithm: HS256
      authority-claim-name: roles
```

### How do I configure OIDC (OAuth2 Resource Server)?
Use `type: OIDC` and provide the Keycloak/IdP `issuer-uri` and `jwk-set-uri`.

**YAML Configuration:**
```yaml
middleware:
  security:
    type: OIDC
    oauth2:
      issuer-uri: https://issuer.example.com/realms/demo
      jwk-set-uri: https://issuer.example.com/realms/demo/protocol/openid-connect/certs
      authorities-claim-path: $.realm_access.roles
```

**Constraints:**
- Unlike `API_KEY` configurations, authentication errors generated by `JWT` or `OIDC` use the standard Spring Security `401 Unauthorized` filter entries, meaning they DO NOT natively use the unified JSON `ErrorMessage` formatting unless wrapped manually.
